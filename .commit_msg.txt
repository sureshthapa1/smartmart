fix: undefined _logger in email_service.py (same bug class as dashboard/routes.py)

A remote commit added a welcome-email feature (send_welcome_email() +
email/welcome.html template) and used _logger.info(...)/_logger.warning(...)
for logging, but this file already has a differently-named logger instance
(logger = logging.getLogger(__name__), no underscore) and _logger was never
defined. This is the same class of bug fixed earlier in dashboard/routes.py:
a logging call added without checking the actual variable name already in
use in the file.

Impact: on both the success and failure path of send_welcome_email(), the
NameError would fire immediately -- masking whatever the original mail-send
outcome was, and (since this is inside a try/except Exception in the
failure path) potentially swallowing the real error silently depending on
call-site handling.

Fixed by using the existing 'logger' variable, consistent with the rest of
the file. Verified with flake8 --select=F821 across the entire codebase:
0 undefined names anywhere.

NOTED, NOT ACTED ON: send_welcome_email() is fully implemented but not
called from anywhere in the codebase yet (checked via grep across all
.py files) -- the registration flow doesn't currently trigger it. Looks
like a staged feature. Not wiring it in without being asked, since I don't
know the intended trigger conditions (e.g. should registration fail if the
welcome email fails to send? what if the customer has no email on file?).

187 tests pass; flake8 clean; pip-audit clean.
